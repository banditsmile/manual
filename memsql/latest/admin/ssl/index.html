SSL Network Encryption¶

As of version 4, MemSQL supports SSL encryption over the network. The configuration closely follows and is compatible with MySQL’s SSL features, including the standard MySQL client and drivers. The encryption library used is OpenSSL.

It’s important to note that enabling SSL encryption between the client and the MemSQL cluster is separate from enabling encrypted communications between nodes inside the cluster itself. The first section in this document is Enabling Client SSL Encryption. The second deals with testing SSL from the client, and the third is Enabling Cluster SSL Encryption.

Feature Note

This MemSQL feature is only available in MemSQL Enterprise Edition. MemSQL Enterprise Edition includes 24x7 support and several enterprise-only features for critical deployments. For more information about MemSQL Community and MemSQL Enterprise, see the MemSQL Editions page.

Enabling Client SSL Encryption¶

You can check to see whether SSL configured by running SHOW VARIABLES:

memsql> show variables like '%ssl%';
+---------------+-------+
| Variable_name | Value |
+---------------+-------+
| have_openssl  | OFF   |
| have_ssl      | OFF   |
| ssl_ca        |       |
| ssl_capath    |       |
| ssl_cert      |       |
| ssl_cipher    |       |
| ssl_key       |       |
+---------------+-------+
7 rows in set (0.01 sec)

To enable SSL, you must generate (or use existing) certificates and keys for the server and client as well as a certificate authority, also known as a CA cert.

Generating SSL Certificates¶

Here are basic steps for generating a set of certificates and keys to use with MemSQL. This process generates only “self-signed” certificates. More sophisticated X509 certificate chains are beyond the scope of this document.

On the Master Aggregator, run these commands to generate the certificates.

cd /var/lib/memsql
mkdir certs
cd certs

## The subject string for certificate signing requests.
## Edit the details to match your organization.
SUBJ="/C=US/ST=CA/L=San Francisco/O=MemSQL/CN="
CA_SUBJ="${SUBJ}memsql.ssl.test.ca"
SERV_SUBJ="${SUBJ}memsql.ssl.test.server"
CLIENT_SUBJ="${SUBJ}memsql.ssl.test.client"

## create the CA cert and key
openssl genrsa 2048 > ca-key.pem
openssl req -new -x509 -nodes -days 3600 -key ca-key.pem -out ca-cert.pem -subj "$CA_SUBJ"

## create the server cert, key, and sign with CA
openssl req -newkey rsa:2048 -days 3600 -nodes -keyout server-key.pem -out server-req.pem -subj "$SERV_SUBJ"
openssl rsa -in server-key.pem -out server-key.pem
openssl x509 -req -in server-req.pem -days 3600 -CA ca-cert.pem -CAkey ca-key.pem -set_serial 01 -out server-cert.pem

## create the client cert, key, and sign with CA
openssl req -newkey rsa:2048 -days 3600 -nodes -keyout client-key.pem -out client-req.pem -subj "$CLIENT_SUBJ"
openssl rsa -in client-key.pem -out client-key.pem
openssl x509 -req -in client-req.pem -days 3600 -CA ca-cert.pem -CAkey ca-key.pem -set_serial 01 -out client-cert.pem

## verify the certificate chain
openssl verify -CAfile ca-cert.pem server-cert.pem client-cert.pem

To use these new certificates, copy the ca- and server- PEM files to all hosts in the cluster. Edit the memsql.cnf file on all aggregators to add the certificate paths in the [server] section as shown below. Then, restart the memsql service on all aggregators.

[server]
basedir = .
... [snip] ...

; These can be absolute paths, or relative to the memsqld binary.
ssl-ca = ./certs/ca-cert.pem
ssl-cert = ./certs/server-cert.pem
ssl-key = ./certs/server-key.pem

Once restarted, check that SSL is now enabled on the aggregators:

memsql> show variables like '%ssl%';
+---------------+---------------------------------------+
| Variable_name | Value                                 |
+---------------+---------------------------------------+
| have_openssl  | ON                                    |
| have_ssl      | ON                                    |
| ssl_ca        | /var/lib/memsql/certs/ca-cert.pem     |
| ssl_capath    |                                       |
| ssl_cert      | /var/lib/memsql/certs/server-cert.pem |
| ssl_cipher    |                                       |
| ssl_key       | /var/lib/memsql/certs/server-key.pem  |
+---------------+---------------------------------------+
7 rows in set (0.00 sec)

Connecting to MemSQL via SSL¶

Copy ca-cert.pem, client-cert.pem, and client-key.pem to your client machine. These are fed in as parameters to the MySQL client. The status command outputs connection details.

$ mysql -uroot -h1.2.3.4 --ssl-ca=ca-cert.pem -e 'status'

Requiring SSL Encryption¶

By default, SSL is not used by the client to connect to MemSQL even if SSL is enabled on the aggregators. To restrict access to clients to SSL only, you must add the REQUIRE SSL clause to the user’s GRANT statement.

memsql> grant all on *.* to 'ssl_user'@'%' REQUIRE SSL;

To connect, the client will need to pass in a certificate that is valid under the server’s CA certificate:

## this fails:
$ mysql -ussl_user -h1.2.3.4 -e "select now()"

## this works:
$ mysql -ussl_user -h1.2.3.4 --ssl-ca=ca-cert.pem -e "select now()"
+---------------------+
| now()               |
+---------------------+
| 2014-04-18 19:34:23 |
+---------------------+

Enabling Cluster SSL Encryption¶

The steps to enable intracluster secure communications is similar to enabling it between SQL clients and the aggregators. Assuming you’ve created a valid set of certificates, copy the ca- and server- PEM files to all leaves and aggregators in the cluster.

Edit the memsql.cnf file on all the nodes to include the local paths to the certificates, as shown below:

[server]
basedir = .
... [snip] ...

; These can be absolute paths, or relative to the memsqld binary.
ssl-ca = ./certs/ca-cert.pem
ssl-cert = ./certs/server-cert.pem
ssl-key = ./certs/server-key.pem

After restarting the cluster, SSL encryption will automatically be used for intracluster communication. If you are sufficiently paranoid you can also add REQUIRE SSL to the GRANT statement of the MemSQL account used for intracluster work. By default, this account is root.

MemSQL 2.6 Release Notes¶

Upgrading to 2.6¶

MemSQL supports various methods for upgrading to version 2.6 depending on you current installation. For more information see: Upgrading MemSQL.

Per-Table Memory Statistics¶

MemSQL 2.6 introduces the INFORMATION_SCHEMA.TABLE_STATISTICS table that gives you detailed, per-table and per-partition memory statistics. For every table in every partition (master, slave, or reference), this table exposes an approximate row count (ROWS) and allocated memory in bytes (MEMORY_USE).

By querying INFORMATION_SCHEMA.TABLE_STATISTICS, you can answer questions like the following:

  • How much memory am I using per-table?
SELECT
    DATABASE_NAME,
    TABLE_NAME,
    SUM(MEMORY_USE)/(1024*1024) AS TOTAL_MEMORY_MB
FROM INFORMATION_SCHEMA.TABLE_STATISTICS
GROUP BY 1, 2
  • Is the data in my tables being distributed more or less evenly around the partitions, or is there skew?
SELECT
    DATABASE_NAME,
    TABLE_NAME,
    MIN(ROWS),
    MAX(ROWS),
    FLOOR(AVG(ROWS)) AS avg_rows,
    ROUND(STDDEV(ROWS)/AVG(ROWS),3) * 100 AS row_skew,
    MIN(MEMORY_USE),
    MAX(MEMORY_USE),
    FLOOR(AVG(MEMORY_USE)) AS avg_memory,
    ROUND(STDDEV(MEMORY_USE)/AVG(MEMORY_USE),3) * 100 AS memory_skew
FROM INFORMATION_SCHEMA.TABLE_STATISTICS
GROUP BY 1, 2
HAVING SUM(ROWS) > 10000
  • How much memory am I using for reference tables on the leaves?
SELECT
    SUM(MEMORY_USE) / (1024*1024) AS TOTAL_MEMORY_MB,
    SUM(ROWS) AS TOTAL_ROWS
FROM INFORMATION_SCHEMA.TABLE_STATISTICS
WHERE NODE_TYPE="Leaf" AND PARTITION_TYPE="Reference"

Optimizer and Query Execution Improvements¶

MemSQL 2.6 offers several performance improvements across the board. Some highlights include the following:

  • Faster JSON expression performance by optimizing shallow copy code paths and significantly reducing memory use for JSON expression evaluation.
  • Optimizations for MIN and MAX aggregates used on the inner table of a join. MemSQL 2.6 speeds up queries like SELECT MAX(clicks.time) FROM users INNER JOIN clicks ON users.id = clicks.user_id WHERE users.id = 10.
  • Higher defaults for max_allowed_packet and bigger default LOAD DATA batches improve out-of-the-box data loading performance.

Distributed Improvements¶

MemSQL 2.6 includes manageability improvements for the distributed system:

  • SHOW PARTITIONS now includes a Locked column that indicates whether a partition is locked. Partitions are locked for metadata-switch operations during the PROMOTE phase of REBALANCE as well as briefly during BACKUP.
  • High availability is now enabled for clusters with two aggregators. MemSQL uses a quorum algorithm to determine whether the majority of aggregators are reachable, to protect against both machine failure and network partitions. Before MemSQL 2.6, if the child aggregator in a set of two lost contact with the master, it would fail to reach a majority and shut itself down. After 2.6, the child aggregator will continue to run without a master. This offers high availability for reads and writes, with a small but non-zero chance of reading from the wrong leaf in the case of a network partition. For both high availability and protection against network partitions, we strongly recommend three or more aggregators.
  • REBALANCE is now more robust for large data partitions.
